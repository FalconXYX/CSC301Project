2.6 Create Search Functionality (Patient search + ranked list)
“As a patient, I want my questionnaire responses to be automatically matched against the database of clinics so that I receive a ranked list of providers who offer the specific services I need within a reasonable distance.”

Independent:
Can be built without Clinic Management UI. This was made with clinics/services already existing in the database from previous sprint. This can be worked on in any order

Negotioable:
User story is negotioable as specific details in implementation are not specified. The user story is open to interpretation. And a concrete solution can be agreed upon and subissues can be created based on discussion.

Valuable to the User or Customer:
Direct user value as it allows patients to find physiotherapists based on their needs and location.

Estimatable:
Yes, 5 story points are assigned to this user story suggesting it would take around 5 development hours. Refer to UserStoryExample_1.png.

Testable:
Yes, the user story is testable as it can be tested through user interface and API testing. 

Understandable:
Yes, the user story is understandable as it is written in a simple and easy to understand language.

Yes, the user story is prioritized as P0 as it is a high priority user story.

2.4 Clinic Management Front End
“As a Clinic Manager, I want to be able to create and manage my clinic’s public profile so that it is easily searchable and provides patients with detailed information beyond just the basics.”

Independent:
Can be built without Clinic Management UI. This was made with clinics/services already existing in the database from previous sprint. This can be worked on in any order

Negotioable:
User story is negotioable as specific details in implementation are not specified. The user story is open to interpretation. And a concrete solution can be agreed upon and subissues can be created based on discussion.

Valuable to the User or Customer:
Provides direct value to clinic managers by allowing them to control their public presence. It also indirectly benefits patients by improving the quality and completeness of searchable clinic information.

Estimatable:
Yes, 6 story points are assigned to this user story suggesting it would take around 5 development hours. Refer to UserStoryExample_1.png.

Small and Specific:
The story is focused specifically on clinic profile management. While it contains multiple subissues, each subissue is scoped to a clear functionality (e.g., route protection, edit details, image upload), keeping the work manageable within a sprint

Testable:
Yes, the user story is testable as it can be tested through user interface and API testing. For example Creating a clinic profile saves correct data, Editing updates the database, etc.

Understandable:
Yes, the user story is understandable as it is written in a simple and easy to understand language. 

Prioritized:
Yes, the user story is prioritized as P1 as it is important for user.

Ready for Sprint Planning:
Yes, the user story is ready for development as it is not blocked by any other user story.
